{oA.Reverse}

Migrate the dataflow or dataviz layer to the package

Data storage or data visualization technologies have accumulated over time. Some of these technologies are outdated, or are not well ported to the Cloud. You have to migrate.

Unfortunately, the volumes and intrinsic complexity of certain technologies make these projects excessively complex and expensive. We offer solutions to automate these migrations. We implement them on a fixed price basis.

Migrate cloud storage technologies automatically

The problem

Large companies have been accumulating storage technologies forever. There is continued stacking, as removing one technology often poses too much risk.

The skills associated with it are becoming rarer and retro-documentation is rarely in place. When companies start, they are consultancy projects, long, expensive and hazardous.

Our solutions

We think it is better to automate the process. To do this, we have developed a unique methodology to automate these migrations based on the unique introspection capabilities of {openAudit} systems.

• {openAudit} will “parse” the source code, it will break down all the complexity of the code using a grammar allowing exhaustive and ultra-granular analyses. All the subtleties will be taken into consideration.

• {openAudit} deduces the overall kinematics and intelligence, which will be reconstructed in an agnostic algorithmic tree. On this basis, {openAudit} will produce “standard SQL”.

• Then the intelligence will be reconstructed at a minimum in the specific SQL of the target database (for example BigQuery for Google, Redshift for Amazon, Azure SQL for Microsoft, etc.).

• All complex processing that cannot be reproduced in simple SQL will be controlled by an executable (Node JS or other). Typically “For Loop” cursors, variables, “If Else ” conditional code , “ Switches ”, procedure calls, etc.

• {openAudit} produces “Yaml” files (intuitive files). Thus, the understanding of complexity is shared with as many people as possible.

• Eventually, new orchestration mechanisms can be implemented, to deconstruct cursors of cursors (loops of loops) to optimize transformation chains, etc.

Migrate data visualization technologies to the Cloud in an automated way

The problem

Dataviz solutions embed more and more intelligence. And some of them have been there for several decades. There are thus an innumerable number of carefully constructed management rules which threaten not to be carried over into the target technology in the event of a failure of the migration.

And the professions want “1 for 1”, without the slightest regression, with an almost identical result. To do this, service companies offer long, expensive and hazardous service missions.

Our answers

{openAudit}, relying on its exceptional introspection capabilities of the source data visualization layer, makes it possible to automate the process of “rebuilding” the platform in the target technology.

Methodology



• Most dataviz tools have two things in common: a semantic layer which forms the interface between IT and the business and a dashboard editor.

• We leverage {openAudit}'s automated reverse engineering to fully deconstruct the complexity at source, allowing us to address it again in the target technology.

• {openAudit} will be able to feed the target technology from the single semantic layer, a sort of “pivot model” (for certain technologies). This model will have been automatically generated from the data visualization tools to be decommissioned,

• The structure of the initial dashboard will also have been analyzed by {openAudit}, and it can also be transcribed into the target technology.
