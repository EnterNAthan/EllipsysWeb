{oA.Lineage}

Cartographier Un Système d’Information

Les systèmes d'information sont d'une telle complexité, que les équipes ne parviennent plus à naviguer efficacement dans les flux pour opérer les investigations souhaitées. Les architectures Data Mesh orientées Domaines accélèrent cet état de fait.

Le data lineage technique et multi-technologique de {openAudit} permet d'appréhender l'origine et les usages de chacune des données des systèmes on-premise et/ou Cloud d'un clic de souris. Les analyses sous-jacentes sont rejouées quotidiennement.

Faire du Data Lineage dans les Dataflows

Au départ de n’importe quel « data point » du Système d’Information (un champ d’une table, une table, un fichier, un schéma, etc.), {openAudit} permet d'appréhender son origine, son devenir et ses usages grâce au data lineage technique, multi technologique. Le process est dynamique, opéré en mode delta, et sans rupture : {openAudit} traite les vues, les procédures dynamiques, etc.

Une analyse d’impact de bout en bout

{openAudit} propose une vue qui permet de connaître instantanément l’ensemble des usages d’un data point dans la couche de data visualisation, en faisant abstraction de toutes les transformations.

La Data Observabilité, Buzzword ou nécessité ?

Le data lineage de {openAudit} permet de sensiblement renforcer l’observabilité des données, un sujet de préoccupation fort pour les entreprises.

BCBS 239 : L'enjeu de la fréquence et de l'exactitude du reporting de risque.

BCBS 239 est un texte bancaire relatif à la qualité du reporting de risque. Il impose la possibilité de sourcer les données de risque dans les systèmes avec une solution de data lineage.

Cybersécurité : le data lineage technique, l’arme idéale pour la Data Leaks

La DLP (Data Leaks Prevention) est une discipline de la cyber sécurité qui vise à détecter les fuites de données sensibles. Le data lineage technique couplé à la détection des usages de l’information permet de savoir au départ d’une donnée de risque, qui y accède, et dans quel contexte.

Faire du Data Lineage dans la couche de Data visualisation

Le problème

Certaines technologies de data visualisation utilisent des couches sémantiques pour créer de l'intelligibilité pour le métier, et ainsi lui donner de l'autonomie. Mais ces couches sémantiques créent de l'abstraction : les champs physiques sous-jacents sont difficiles à identifier, ce qui rend le sourcing complexe.

De plus, les technologies de data visualisation interrogent souvent des vues, des vues de vues… ce qui complique encore une fois le sourcing.

Par ailleurs, les technologies de data visualisation permettent maintenant de faire de la "data prep" dans des proportions importantes : l’intelligence s’est déplacée des datamarts vers les solutions de data visualisation. Cela créé une forte opacité avec des règles de gestion qui ont tendance à foisonner.

Nos réponses

{openAudit} permet d'appréhender l'ensemble des technologies de data visualisation de l'entreprise sur une interface unique d'analyse d'impact. Le data lineage dans le dashboard ou dans les couches alimentation peut être déclenché depuis cette interface.

Ses spécificités



• {openAudit} opère un data lineage dans les expressions, dans les variables, etc., pour identifier les champs directement ou indirectement en source d'un data point.

• {openAudit} analyse le contenu des vues pour identifier les champs physiques qui sont en source des données d'une couche de data visualisation, même si les vues sont empilées.

• {openAudit} combine les analyses des différentes technologies de dataviz dans une même grille, ce qui permettra aux métiers et à l'IT de réaliser des analyses d'impact entre toutes les couches d'alimentation et tous les outils de data visualisation.
