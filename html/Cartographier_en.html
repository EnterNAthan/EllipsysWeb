{oA.Lineage}

Mapping an Information System

Information systems are so complex that teams are no longer able to effectively navigate the flows to carry out the desired investigations. In recent times, domain-oriented Data Mesh architectures have accelerated this state of affairs.

Our multi-technology data lineage, coupled with the analysis of data usage, makes it possible to map systems precisely and continuously.

Do Data Lineage in the Dataflows

The problem

We meet companies which have just migrated to the Cloud and which are already suffering from hypertrophy of their Information System, with entangled flows, replication, etc.

Data teams spend far too much time trying to master their Information Systems. When architectures are older, experts are often missing to reverse document Information Systems and enable real knowledge sharing.

Our solutions

Starting from any “data point” (a field in a table, a table, a file, a diagram), {openAudit} allows us to understand its origin, its future and its uses, through a data lineage technical, multi-technological.

The process is dynamic, operated in delta mode, daily, and it is therefore perfectly synchronized with the Information System.

We offer different levels of analysis



• Point cloud: this view allows you to instantly know the uses (in the reporting layer, via queries) of a data point, ignoring transformations.

• Mapping: this view allows from any data point (field, table, database schema) to display a complete map of the upstream or downstream flow, that is to say from operational sources to to the exposure of data (dataviz, queries). The information used is highlighted and the uses of the information are specified upon overview: who consults the data, when, how.

• Granular data lineage: this view allows you to progressively follow the deployment of data in the Information System from a data point through iterative clicks, or conversely, to go back to operational sources. Each transformation (ELT/ETL job, procedural/object code) can be analyzed with “drill through”. The precise details of the uses of the data (who consults it, when, how, etc.) are specified.



{openAudit} resolves all breaks in the data lineage



• Views: if they are stored, {openAudit} will read them, even if they are stacked (views of views of views...).

• Dynamic SQL: If {openAudit} fails to resolve directly, dynamic SQL is resolved with execution parameters, or with execution logs.

• Others: in the case of information transfer by FTP, or when the database schema is not specified (this is the case in many ETLs), {openAudit} resolves these breaks by structural recognition, or or {openAudit} } reads the Batch / Shell.

• {openAudit} analyzes all processing technologies (object/procedural language, ELT/ETL), on-premise or Cloud, and combines them in a single data flow, at the finest level. Drill through provides access to the code.


Uses cases

Data Observability, Buzzword or necessity?

{openAudit}'s data lineage significantly strengthens data observability, a major concern for businesses.

BCBS 239: The issue of frequency and accuracy of risk reporting.

BCBS 239 is a banking text relating to the quality of risk reporting. It requires the possibility of sourcing risk data in systems with a data lineage solution.

Cybersecurity: technical data lineage, the ideal weapon for Data Leaks

DLP (Data Leaks Prevention) is a cyber security discipline that aims to detect leaks of sensitive data. Technical data lineage coupled with the detection of information uses makes it possible to know at the start of risk data, who is accessing it, and in what context.

Do Data Lineage in the data visualization layer

The problem

Some data visualization technologies use semantic layers to create intelligibility for the business, and thus give it autonomy. But these semantic layers create abstraction: the underlying physical fields are difficult to identify, which makes sourcing complex.

In addition, data visualization technologies often query views, views of views, etc., which once again complicates sourcing.

Furthermore, data visualization technologies now make it possible to do "data prep " in significant proportions: intelligence has moved from data marts to data visualization solutions. This creates a strong opacity with management rules that tend to abound.

Our solutions

{openAudit} allows you to understand all of the company's data visualization technologies on a single impact analysis interface. Data lineage in the dashboard or in the power layers can be triggered from this interface.

Its specificities



• {openAudit} performs data lineage in expressions, variables, etc., to identify fields directly or indirectly at the source of a data point.

• {openAudit} analyzes the content of views to identify the physical fields that are the source of data for a data visualization layer, even if the views are stacked.

• {openAudit} combines the analyzes of the different dataviz technologies in the same grid, which will allow business and IT to carry out impact analyzes between all feed layers and all data visualization tools.
